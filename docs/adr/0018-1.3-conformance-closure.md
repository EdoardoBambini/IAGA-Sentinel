# ADR 0018: Chiusura di conformità 1.3 (flag is_authoritative, attributi OTel, env scrub, feature verify-only)

- **Status**: Accepted
- **Date**: 2026-06-08
- **Deciders**: Edoardo Bambini
- **Milestone**: 1.3.1
- **Relates to**: ADR 0015 (verifier standalone), ADR 0016 (OTel receipt export), ADR 0006 (kernel MVP), ADR 0012 (drift replay additivo)

## Contesto

La revisione di giugno 2026 della roadmap attribuisce alla minor 1.3 ("verifier sovereignty") un insieme di primitive OSS che la 1.3.0 ha spedito solo in parte. Un confronto diretto codice-contro-documento ha isolato quattro scarti onesti, tutti chiudibili in modo additivo:

1. Lo span OpenTelemetry della ricevuta non espone gli attributi citati dalla roadmap (`iaga.receipt.id`, `iaga.chain.head`, `iaga.policy.verdict`): usava solo chiavi `receipt.*` in camelCase.
2. La ricevuta non portava il flag esplicito `is_authoritative=false` promesso "in ogni receipt OSS": l'onestà sull'enforcement soft viveva solo come metodo del kernel (`UserspaceKernel::is_authoritative()`), non dentro l'oggetto-prova.
3. L'hardening dell'ambiente del `UserspaceKernel` si fermava a un allowlist di 11 variabili ereditate, senza il denylist di variabili sensibili note espandibile via TOML.
4. La ricetta di build riproducibile `--no-default-features --features verify-only` non funzionava perché la feature `verify-only` non esisteva.

Il quinto scarto, il verificatore multilingue TypeScript/Python, è grande e indipendente: non rientra in questa patch, va nella 1.3.2. L'audit ha inoltre rilevato che le feature 1.3 `otel-receipts` e `plugin-manifest-signing` non erano coperte dalla CI (clippy e test girano a feature di default).

## Decisioni

### 1. Flag `is_authoritative` nella ricevuta

`ReceiptBody` guadagna un campo opzionale `is_authoritative: Option<bool>` con `#[serde(default, skip_serializing_if = "Option::is_none")]`. La pipeline OSS lo popola a `Some(false)` perché nessun kernel autoritativo è spedito nel build community (`UserspaceKernel` è soft, `BpfKernel` è scaffold). Quando assente è eliso da `signing_bytes`, quindi le ricevute prodotte prima della 1.3.1 restano byte-identiche e verificano invariate. Un build Enterprise con kernel eBPF/LSM autoritativo imposterà `Some(true)`.

### 2. Attributi OTel della roadmap

`emit_receipt_span` aggiunge `iaga.receipt.id` (l'identità stabile `run_id:seq`), `iaga.chain.head` (l'hash del corpo della ricevuta, cioè la nuova testa di catena), `iaga.policy.verdict` e `iaga.is_authoritative`. Le chiavi `receipt.*` esistenti restano come alias di retrocompatibilità. L'allineamento completo alle convenzioni semantiche GenAI (`gen_ai.*`, traces/metrics/logs separati) resta una consegna della 1.4.

### 3. Scrub dell'ambiente sensibile + override TOML

`UserspaceKernel` applica un denylist di 23 variabili note che portano segreti (credenziali cloud e provider di modelli, token di registry, il path della chiave di firma delle ricevute) sopra l'allowlist esistente. Lo scrub è autoritativo: le variabili note non raggiungono il figlio governato neanche se passate esplicitamente in `ProcessSpec.env`. Il denylist è estendibile a runtime con un file TOML indicato da `IAGA_SENTINEL_ENV_DENYLIST` (schema `deny = [...]`); un file illeggibile o malformato degrada al built-in con un warning, mai un crash sul percorso di lancio.

### 4. Feature `verify-only`

`iaga-sentinel-verify` dichiara `[features] default = ["verify-only"]`, `verify-only = []`. La feature non tira dipendenze e non gating: esiste perché la ricetta `cargo build --release --no-default-features --features verify-only` sia valida e stabile tra le release (ADR 0015). Build normale e ricetta esplicita producono lo stesso binario.

### 5. Copertura CI delle feature 1.3

Il job `test` aggiunge due step: `cargo test -p iaga-sentinel-core --features otel-receipts --lib` e `--features plugin-manifest-signing --lib`. Erano le due primitive 1.3 senza copertura CI: il loro codice (incluso il test gated `receipt_span_carries_receipt_attributes`) non veniva compilato né eseguito a feature di default.

## Conseguenze

### Positive

- La 1.3 spedita corrisponde alla 1.3 dichiarata dalla roadmap (meno il verificatore multilingue, esplicitamente rinviato alla 1.3.2).
- L'onestà sull'enforcement soft è ora dentro la ricevuta, non solo nella documentazione.
- Un segreto noto nell'ambiente dell'host non raggiunge un agente governato, con due strati (allowlist + denylist) e un punto di estensione operativo via TOML.

### Negative

- `is_authoritative` partecipa alla firma: una ricevuta 1.3.1 non è byte-identica a una 1.3.0 dello stesso evento. La 1.3.0 resta verificabile per sempre, il campo è additivo ed eliso quando assente.
- Lo scrub autoritativo può rimuovere una variabile che un chiamante intendeva passare: per scelta, i segreti vanno consegnati da un canale controllato, non dall'ambiente di processo.

### Neutre

- `verify-only` è un marker: oggi non gating nulla, ma stabilizza la ricetta di build riproducibile.
- L'allineamento `gen_ai.*` completo e il verificatore multilingue restano fuori, rispettivamente in 1.4 e 1.3.2.
