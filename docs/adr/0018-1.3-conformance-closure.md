# ADR 0018: 1.3 Conformance Closure

- **Status:** Accepted
- **Date:** 2026-06-08

## Context

The 1.3 line added verifier-focused primitives. A final review found a few small gaps between the public docs and the implementation.

## Decision

Close the 1.3 gaps with additive changes:

- Add optional `is_authoritative` to `ReceiptBody`; the open build records `false`.
- Add roadmap-named OpenTelemetry keys alongside existing receipt span attributes.
- Harden `UserspaceKernel` environment handling with a built-in denylist for known secret-bearing variables.
- Allow operators to extend the denylist with `IAGA_SENTINEL_ENV_DENYLIST`.

When `is_authoritative` is absent, older receipts continue to verify. New 1.3.1 receipts include the explicit posture flag.

## Consequences

The evidence now states the open build's soft-enforcement posture directly. Telemetry names are easier to consume, and governed child processes receive fewer accidental host secrets.

The denylist may remove a variable that a caller expected to pass. That is deliberate: secrets should be delivered through controlled channels, not inherited process environments.
